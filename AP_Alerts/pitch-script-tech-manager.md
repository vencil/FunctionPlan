# 講稿 B：面對技術團隊主管（Engineering Manager / VP Infra / CTO）

> 預計時長：5 分鐘。語氣：講成本、講風險、講時間線，技術點到為止。

---

## 開場（30 秒）

我想用三分鐘解釋一個問題：為什麼你的監控系統會隨著業務規模增長而變成負擔，以及我們怎麼讓它不再是負擔。

---

## 第一段：現在的成本結構（1 分鐘）

多租戶監控有一個隱藏的擴展性陷阱。

每加一個租戶，就要複製一整套警報規則。100 個租戶 × 35 條規則 = 3,500 條規則同時在跑。這帶來三個成本：

**運算成本：** Prometheus 每 15 秒評估 3,500 條 PromQL，評估時間從 20 毫秒膨脹到 850 毫秒以上，最後要不就是加機器，要不就是警報延遲。

**人力成本：** 每次租戶調閾值都要走 PR 流程。平台團隊的時間不是花在建設，而是花在幫租戶除錯 PromQL 語法。

**風險成本：** 3,500 條規則擠在一起，改一條可能影響其他租戶。沒有稽核、沒有隔離、沒有安全的回退機制。

---

## 第二段：我們帶來什麼改變（1.5 分鐘）

我們用一個架構改動，把上面三個成本壓平。

**規則數固定。** 不管 10 個還是 1,000 個租戶，Prometheus 只跑 85 條規則。加租戶只是多一個 YAML 檔案，不是多一套規則。實測 100 租戶，評估時間依然是 20 毫秒。

**租戶自助。** 租戶不需要學 PromQL。他們寫一行 YAML — 比如 `mysql_connections: "80"` — 就完成了。我們有互動式工具幫他們產生配置，五分鐘搞定。

**完整生命週期。** 從導入、日常營運、到最後的下架，每個環節都有工具。每個工具都有 dry-run 模式，所有操作可回退。

這不只是一套監控工具，而是一套讓監控的維護成本不隨租戶數增長的治理方案。

---

## 第三段：遷移風險（1 分鐘）

你可能最關心的是：「我們現在有幾千條舊規則，遷移會不會炸？」

答案是不會。原因有三：

**一、不強迫一次切換。** 我們的工具會先幫你分類所有舊規則，輸出一份報表讓你的團隊逐條決策。

**二、新舊規則並行。** 轉換後的規則在命名空間上跟現有規則完全隔離。在切換前，我們用 API 逐一比對新舊規則的輸出值，確認完全一致才切。

**三、隨時可退。** 我們的架構基於 Kubernetes 原生機制，任何一個規則包都可以一條命令卸載，不影響其他規則，Prometheus 不會崩潰。

---

## 第四段：可驗證性（30 秒）

最後一點。我們內建了一個真實負載的端對端展演。不是改閾值假裝觸發，而是真的注入連線風暴和 CPU 壓力，看警報觸發，然後清除負載看警報自動消失。整個循環不到五分鐘，可以在任何評估場景中現場執行。

```
make demo-full
```

一條命令，肉眼可見。

---

## 收尾（30 秒）

總結三個數字：

- **85** — 不管多少租戶，規則數固定 85 條
- **0** — 租戶需要寫的 PromQL 行數
- **5 分鐘** — 從零開始跑完整個 demo 的時間

我們的 Dev Container 環境是開箱即用的，如果團隊有興趣評估，隨時可以上手。

---

*主管常問的追加問題：*

- **「導入時間線？」** → 新租戶：5 分鐘（互動式工具）。遷移既有規則：依規模，100 條以內一天，1000+ 條建議一週分批
- **「需要改現有 Prometheus 嗎？」** → 不需要。加一個 exporter + 掛載規則包，不動現有設定
- **「團隊學習曲線？」** → 租戶零學習（只寫 YAML）。平台團隊需要理解向量匹配原理，有完整架構文件
- **「授權模式？」** → MIT 開源
- **「跟 Datadog / Grafana Cloud 的定位差異？」** → 我們不替代 Prometheus，而是解決多租戶場景下的規則管理問題。可以搭配任何 Prometheus 相容的上層平台使用
